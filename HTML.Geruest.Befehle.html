Git-Befehle

git config

git init

git add

Mit dem Befehl git add kann ein Repository (also ein Verzeichnis) unter die Verwaltung durch Git gestellt werden. Nur die Dateien, die mit Git add zur Verwaltung hinzugefügt wurden werden beim nächsten Commit gespeichert. Wichtig ist, dass später beim commit der Stand gespeichert wird, den die Datei beim Eingeben des Befehls git add hatte. Also sollte git commit immer unmittelbar nach git add verwendet werden.

git remote add

Mit dem Befehl $git remote add wird ein lokales Repository z.B. einem externen Repository hinzugefügt.
Hierzu werden folgende Dinge benötigt:
Ein Remote Name, z.B. von der Herkunft
Eine entfernte URL, z.B. https://github.com/Superscoti/Github_Projekt.git

git commit

Unter einem Commit versteht man einen Schnappschuss des aktuellen Stand aller Dateien. Wurde eine Datei im Repository nicht geändert, wird für diese kein Schnappschuss erstellt. Stattdessen wird der vorherige Schnappschuss weiterhin verwendet. Die einzelnen Commits können später wieder aufgerufen werden.

Bei jedem Commit kann ein kurzer Kommentar hinzugefügt werden. Dieser Kommentar dient zur Übersicht und hilft Änderungen nachzuverfolgen und wiederzufinden. Ein commit kann über das Terminal durch den Befehl $git add (Dateiname) und $git commit "Short Message" erstellt werden. In der GitHub Desktop Applikation kann ein Commit unter dem Reiter Changes durch den Click auf den Button Commit to master ausgeführt werden.

Optional kann ein Commit auch in dem Programm in dem die Änderungen durchgeführt wurden gemacht werden. Vorausgesetzt das Programm unterstützt git (z.B IntelliJ, Visual Studio Code,...).

git branch

Branches sind eine Art Verzeigung und werden auch Versionszweige genannt. Sie werden benutzt damit mehrere Personen gleichzeitig an einem Projekt arbeiten können. Dadrch können zwei verschiedene Probleme, Funktionen oder sonstige Inhalte in einem Projekt parallel gefixt oder bearbeitet werden. Beim Erstellen eines neuen Repositories entsteht als Standard der „master“ Branch.

Mit dem Befehl git branch [branchname] kann ein neuer Branch erstellt werden. Wenn die Arbeit auf einem Branch erledigt wurde, kann dieser durch mergen wieder mit dem „master“ zusammengeführt werden. Anschließend kann der Branch mit dem Befehl git branch –d [branchname] wieder gelöscht werden.

Mit dem Befehl git branch –a können alle Branches aufgelistet werden. Wenn nur die lokalen Branches aufgelistet werden sollen, kann da „-a“ weggelassen werden.

git checkout

Durch den Befehl git checkout [branchname] kann zwischen den einzelnen Verzweigungen hin und her gewechselt werden. So kann der Branch angesprochen werden, auf dem man selbst arbeitet. Git ändert hier bei einem checkout die Quelldatei. Das Ganze funktioniert im Normalfall aber nur, wenn der Branch bereits commited wurde oder nichts darin verändert wurde. Für das Erstellen eines Branches und das Auschecken dieses gibt es folgenden Befehl, der beides auf einmal macht: git checkout –b [branchname].

git merge

Mit $git merge kann man einzelne Änderungen von einem Branch einem anderen Branch zuweisen. Allerdings können hierbei viele Fehler entstehen. Diese werden als unmerged angezeigt, wenn man $git status in der Konsole abfragt. $git status kann auch aufzeigen, um welche Art von Fehler es sich handelt. Um den Merge abzuschließen, müssen die Konflikte behoben, die Marker entfernt, die Dateien dem Index hinzugefügt und die Änderungen committet werden.

Beispiele für Fehler sind:
error: Entry fileName not uptodate. Cannot merge. (Changes in working directory)
error: Entry fileName would be overwritten by merge. Cannot merge. (Changes in staging area)
https://www.atlassian.com/git/tutorials/using-branches/merge-conflicts

Git merge Optionen sind:
--no-ff Hier wird ein merge Commit erstellt, obwohl ein schneller Vorlauf ebenso möglich wäre
--squash Hier werden alle integrierten Änderungen zusammengefügt. Sozusagen als Paket erhalten, anstatt als einzelne Pakete
--abort Hier wird bei einem Konflikt das Zusammenführen abgebrochen und der Vorzustand wiederhergestellt.

git rebase

Mit dem Befehl git Rebase kann man alle Änderungen, die an einem Branch gemacht wurden, auf einen anderen Branch erneut anwenden. Git geht hierbei auf den letzten gemeinsamen Vorfahren der beiden Branches (den auf dem man arbeitet und den auf den man Rebasen möchte). Dann werden Informationen zu den Änderungen gesammelt, die bei jedem Commit des aktuellen Branches gemacht wurden. Diese Änderungen werden in temporäre Dateien gespeichert und der aktuelle Branch wird auf den gleichen Commit gesetzt, wie der Branch den man Rebasen möchtet. Dann werden alle Änderungen erneut durchgeführt.

git fetch

Der Befehl  git fetch importiert Commits von einem Remote-Repository in das lokale Reposirory. Diese Commits werden als Remote-Branches und nicht als normale, lokale Branches gespeichert. Das bietet Entwicklern die Möglichkeit, Änderungen zu reviewen, ehe sie diese in die eigenen Kopien des Projekts integrieren.

git push

Dieser Befehl wird benötigt, um Änderungen im lokalen Repository an ein entferntes Repository zu übermitteln. Dabei werden die Daten an einen Host übertragen (z.B. GitHub) und andere können diese nun klonen. Bei manueller Eingabe folgen nach git push noch die Parameter „remote_name“ und „branch_name“.

git pull

Wenn du die aktuellsten Änderungen empfangen möchtest und direkt in deine lokale Kopie speichern willst, nutze diesen Befehl. Er bildet sozusagen eine Mischung aus den Befehlen git fetch und git merge.
Solltest du keinen merge wollen, kannst du auch git pull --rebase eingeben.
Für genauere Informationen, was gedownloadet und gemerged wird, verwende git pull --verbose.
Anschließend kannst du mit den neusten Dateiversionen arbeiten.
